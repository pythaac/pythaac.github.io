## **1. DNS란**

![dns](/assets/image/2025-02-11_1.webp)

### **1-1. DNS의 정의**
DNS(Domain Name System)는 **도메인 네임을 해당하는 IP 주소로 변환하는 분산 데이터베이스 시스템**입니다.
예를 들어, `www.google.com`에 접속하려면 실제로는 **Google의 공인 IP(예: `142.250.191.46`)**로 요청해야 합니다.
하지만 숫자로 된 IP를 사람이 직접 기억하기 어렵기 때문에, **DNS를 통해 사람이 입력한 도메인을 IP로 변환**해야 합니다.

### **1-2. DNS가 필요한 이유**
- **IP 주소 대신 기억하기 쉬운 도메인 사용 가능**
- **IP 변경 시에도 도메인은 동일하게 유지** (도메인 소유자는 IP 변경만 DNS 서버에 반영하면 됨)
- **로드 밸런싱, CDN(Content Delivery Network)과 같은 기술을 지원**
- **도메인 네임을 계층적으로 관리하여 인터넷 구조를 효율적으로 운영 가능**

## **2. DNS 서버 찾기**

![dns](/assets/image/2025-02-11_2.webp)

### **2-1. 캐시 (브라우저, 운영체제)**
클라이언트(사용자의 PC 또는 모바일 기기)는 DNS 요청을 보내기 전에 먼저 **캐시(Cache)를 확인**합니다.
DNS 캐시는 여러 단계에서 유지될 수 있습니다.

1. **브라우저 캐시**
   - 웹 브라우저(Chrome, Firefox 등)는 최근 방문한 사이트의 DNS 정보를 일정 기간 저장함.
   - 예: 사용자가 `www.google.com`을 방문했다면, 브라우저가 이를 캐시하여 DNS 요청 없이 바로 접속 가능.

2. **운영체제(OS) 캐시**
   - 브라우저에서 캐시를 찾지 못하면 OS의 DNS Resolver 캐시를 확인함.
   - Windows: `ipconfig /displaydns` 명령어로 확인 가능.

3. **로컬 네트워크 DNS 캐시**
   - 가정용 공유기, 회사 네트워크의 내부 DNS 서버도 캐시를 유지할 수 있음.

### **2-2. 로컬 네트워크 DNS 서버**
로컬 네트워크(사내 네트워크, 학교, ISP 내부망)에는 자체 DNS 서버를 운영할 수 있습니다.
- 조직 내부에서는 자체 DNS 서버를 두어 특정 도메인(예: `company.local`)을 관리함.
- 사내 DNS 서버가 요청을 처리할 수 없는 경우, ISP DNS 서버로 전달함.

### **2-3. ISP DNS 서버**
인터넷 서비스 제공업체(ISP)는 자체 DNS 서버를 운영하며,
- 사용자의 PC가 직접 인터넷의 DNS 서버와 통신하는 대신 **ISP DNS 서버가 중개 역할**을 수행.
- ISP DNS 서버도 캐시를 보유하여 **반복된 요청을 빠르게 처리**할 수 있음.

### **2-4. 루트 DNS 서버**
만약 ISP DNS 서버에서도 요청한 도메인의 IP 정보를 찾을 수 없다면,
**루트 네임서버(Root Name Server)에 질의**하여 최상위 도메인(TLD) 정보를 요청합니다.

#### **DNS 계층 구조**
DNS는 계층적으로 운영되며, 다음과 같이 구성됩니다.
1. **루트 네임서버(Root Name Server)**  
   - 인터넷에서 가장 최상위 계층의 DNS 서버 (전 세계에 13개 존재)
   - TLD(Top-Level Domain, 최상위 도메인) 서버 정보를 반환함.
   
2. **TLD 네임서버(Top-Level Domain Name Server)**  
   - `.com`, `.net`, `.org` 등의 최상위 도메인 정보를 제공.
   - 예를 들어 `www.google.com`을 요청하면 `.com`을 관리하는 TLD 서버로 연결됨.
   
3. **도메인 네임서버(Authoritative Name Server)**  
   - 특정 도메인의 실제 IP 정보를 보유하고 있는 서버.
   - `google.com`의 네임서버(NS Record)를 반환.

## **3. 클라이언트에서 라우터까지**

### **3-1. 클라이언트의 DNS 요청 생성**
DNS 변환 과정에서 캐시에 데이터가 없다면, 클라이언트(PC, 스마트폰 등)는 **DNS 요청 패킷**을 생성합니다.
- **UDP 포트 53**을 사용하여 DNS 쿼리를 생성함.
- 요청 내용: `"www.google.com"의 A 레코드를 조회하는 쿼리`

### **3-2. 라우터의 IP 주소를 얻는 과정**
클라이언트는 **기본 게이트웨이(Default Gateway, 즉 라우터)의 IP 주소**를 알아야 합니다.
- DHCP(Dynamic Host Configuration Protocol)를 통해 **라우터의 IP 주소(예: `192.168.1.1`)를 자동으로 할당받음.**
- 클라이언트는 **라우터를 기본 게이트웨이로 설정하고, 모든 외부 네트워크 요청을 이곳으로 보냄.**

### **3-3. MAC 주소를 활용하여 라우터로 DNS 요청 전송**
- 클라이언트가 IP 패킷을 생성한 후, **이더넷 프레임을 만들어 MAC 주소를 포함**하여 라우터로 보냅니다.
- 클라이언트는 ARP를 통해 라우터의 MAC 주소를 확인합니다.

### **3-4. ARP(Address Resolution Protocol)이란**
ARP는 **IP 주소를 MAC 주소로 변환하는 프로토콜**입니다.
- 클라이언트는 "`192.168.1.1`의 MAC 주소가 뭐야?"라는 ARP 요청을 브로드캐스트로 보냄.
- 라우터는 ARP 응답을 보내 자신의 MAC 주소를 알려줌.
- 클라이언트는 **DNS 요청을 라우터의 MAC 주소로 전송**함.

## **4. 라우터에서 ISP 인터넷 게이트웨이까지**

### **4-1. 라우터가 DNS 서버의 IP 주소를 알고 있는 방법**
라우터는 DNS 서버의 IP 주소를 여러 방법으로 알고 있을 수 있습니다.

1. **DHCP를 통해 ISP로부터 자동으로 할당받음**
   - 가정용 인터넷의 경우, ISP가 제공하는 DNS 서버 주소(예: `168.126.63.1`, `8.8.8.8`)를 DHCP를 통해 라우터에 설정함.
   
2. **수동 설정 (Static Configuration)**
   - 사용자가 직접 `8.8.8.8`(Google DNS) 또는 `1.1.1.1`(Cloudflare DNS) 등의 주소를 설정 가능.

3. **ISP 인터넷 게이트웨이로 요청 (Default Route)**
   - 만약 라우터가 DNS 서버를 찾을 수 없다면, **ISP 인터넷 게이트웨이(기본 게이트웨이)로 패킷을 전송**하여 ISP가 설정한 DNS 서버로 요청을 전달하게 됨.

### **4-2. 패킷에 NAT(Network Address Translation) 적용**
가정용 네트워크에서 라우터는 내부 IP 주소(예: `192.168.1.100`)를 공인 IP(예: `203.0.113.25`)로 변환해야 합니다.
1. **NAT 변환 과정**
   - 클라이언트의 **사설 IP(192.168.1.100) → 공인 IP(203.0.113.25)**로 변경됨.
   - 패킷의 **소스 IP 주소가 라우터의 공인 IP로 변경**되어 ISP 네트워크로 전송됨.

### **4-3. MAC 주소를 활용하여 ISP 인터넷 게이트웨이로 DNS 요청 전송**
1. 라우터는 ISP의 인터넷 게이트웨이 IP 주소(예: `203.0.113.1`)를 알고 있음.
2. **ARP 요청을 통해 게이트웨이의 MAC 주소를 확인**.
3. **DNS 패킷을 게이트웨이의 MAC 주소로 전송**.

## **5. ISP 인터넷 게이트웨이에서 DNS 서버까지**

### **5-1. 라우팅 테이블을 활용하여 ISP 백본 라우터로 DNS 요청 전송**
ISP의 인터넷 게이트웨이는 목적지 IP(예: `8.8.8.8`(Google Public DNS))가 자신의 네트워크 내부에 없음을 라우팅 테이블을 통해 확인한 경우, 다른 ISP 또는 Google 네트워크로 라우팅하기 위해 **라우팅 테이블을 참조하여 패킷을 다음 경로로 전달**합니다.

#### **라우팅 테이블이란?**
- **IP 주소에 따른 최적의 다음 홉(Next Hop) 정보를 저장하는 데이터베이스**.
- ISP의 라우터는 **자신이 아는 네트워크 범위와 해당 목적지를 향하는 최적의 경로를 유지**함.

#### **ISP 백본 라우터까지 전달**
- 라우팅 테이블을 참조하여 패킷을 ISP 내부 네트워크의 백본 라우터로 전달
- 주로 정적 라우팅 또는 내부 라우팅 프로토콜(IGP, OSPF, IS-IS 등)이 활용됨.

### **5-2. ISP 백본 라우터에서 라우팅 테이블/라우팅 프로토콜을 활용하여 DNS 서버의 ISP로 DNS 요청 전송**
- ISP 백본 라우터는 **BGP(Border Gateway Protocol)**를 사용하여 최적의 외부 경로를 찾고,
이를 라우팅 테이블에 반영하여 다른 ISP 또는 목적지 네트워크로 패킷을 전달.
- 즉, BGP를 통해 학습한 경로 정보가 라우팅 테이블에 저장되며, 이를 기반으로 트래픽이 전달됨.
